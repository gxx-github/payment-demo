# WebSocket 心跳机制说明

## 功能概述

为 WebSocket 连接添加了心跳保持机制，确保连接稳定，及时检测连接断开并自动重连。

## 实现细节

### 心跳配置

```javascript
const HEARTBEAT_INTERVAL = 30000; // 每 30 秒发送一次心跳
const HEARTBEAT_TIMEOUT = 10000; // 10 秒内没收到 pong 认为连接断开
```

### 心跳流程

1. **连接建立后立即发送心跳**
   - WebSocket 连接成功后，立即发送第一次心跳消息
   - 启动定时器，每隔 `HEARTBEAT_INTERVAL`（30秒）发送一次心跳

2. **心跳消息格式**
   - 发送：`{"type": "ping"}`
   - 接收：`{"type": "pong"}` 或文本 `"pong"`

3. **超时检测**
   - 每次发送心跳后，设置超时定时器（`HEARTBEAT_TIMEOUT` = 10秒）
   - 如果 10 秒内没有收到 `pong` 响应，认为连接已断开
   - 自动关闭连接（关闭代码：3001，表示心跳超时），触发重连机制
   - **注意**：WebSocket 关闭代码必须是 1000（正常关闭）或在 3000-4999 范围内（自定义代码）

4. **心跳响应处理**
   - 收到 `pong` 响应后，更新最后收到 pong 的时间戳
   - 清除超时定时器，表示连接正常

### 关键组件

#### 定时器引用

```javascript
const heartbeatTimerRef = useRef(null); // 心跳超时定时器
const heartbeatIntervalRef = useRef(null); // 心跳间隔定时器
const lastPongTimeRef = useRef(null); // 最后一次收到 pong 的时间戳
```

#### 核心函数

**`sendHeartbeat()`** - 发送心跳消息
- 检查连接状态（`OPEN`）
- 发送 `{"type": "ping"}` 消息
- 设置超时检测定时器

**`clearHeartbeat()`** - 清理心跳定时器
- 清除所有心跳相关定时器
- 重置时间戳

### 连接生命周期

```
连接建立
  ↓
onopen: 启动心跳（立即发送 + 定时发送）
  ↓
每 30 秒发送 ping
  ↓
收到 pong → 更新时间戳，清除超时定时器
  ↓
如果 10 秒内未收到 pong → 超时检测触发 → 关闭连接 → 重连
  ↓
连接关闭 → 清理心跳定时器
```

## 消息格式

### 发送心跳

```json
{
  "type": "ping"
}
```

### 接收心跳响应

**JSON 格式：**
```json
{
  "type": "pong"
}
```

**文本格式：**
```
pong
```

代码同时支持 JSON 和文本格式的心跳响应。

## 日志输出

心跳相关的日志：

- `💓 发送心跳消息` - 发送心跳时
- `💓 收到心跳响应 (pong)` - 收到 JSON 格式的 pong
- `💓 收到文本心跳响应` - 收到文本格式的 pong
- `⚠️ 心跳超时，连接可能已断开` - 心跳超时警告

## 配置调整

如果需要调整心跳参数，修改文件顶部的常量：

```javascript
// WebSocket 心跳配置
const HEARTBEAT_INTERVAL = 30000; // 每 30 秒发送一次心跳（可根据服务器要求调整）
const HEARTBEAT_TIMEOUT = 10000; // 10 秒内没收到 pong 认为连接断开（建议小于 HEARTBEAT_INTERVAL）
```

### 推荐配置

- **HEARTBEAT_INTERVAL**: 20-60 秒
  - 太小：增加服务器负担，产生不必要的网络流量
  - 太大：无法及时发现连接断开

- **HEARTBEAT_TIMEOUT**: 5-15 秒
  - 建议设置为 `HEARTBEAT_INTERVAL` 的 1/3 到 1/2
  - 需要考虑网络延迟

## 服务器端要求

### 处理心跳消息

服务器需要处理以下心跳消息：

1. **接收 `ping` 消息**
   ```javascript
   if (message.type === 'ping') {
     // 发送 pong 响应
     ws.send(JSON.stringify({ type: 'pong' }));
   }
   ```

2. **或者实现自动心跳**
   - 服务器也可以主动发送心跳，客户端只需响应

### WebSocket 协议支持

某些 WebSocket 实现（如 Socket.IO）提供了内置的心跳机制，如果使用这类库，可以禁用客户端心跳。

## 关闭代码说明

根据 WebSocket 规范，关闭代码（close code）必须遵循以下规则：

- **1000**：正常关闭
- **1001-2999**：**不允许使用**（保留给 WebSocket 规范使用）
- **3000-4999**：自定义关闭代码（可以使用）

心跳超时使用关闭代码 **3001**，表示心跳超时导致的连接关闭。

### 常见关闭代码

- `1000` - 正常关闭
- `3001` - 心跳超时（自定义代码）
- `1001` - **不能使用**（虽然表示"离开"，但不在规范允许的范围内）

## 故障排除

### 问题：WebSocket close 错误（关闭代码无效）

**错误信息：**
```
Failed to execute 'close' on 'WebSocket': The close code must be either 1000, or between 3000 and 4999.
```

**原因：**
使用了不在允许范围内的关闭代码（如 1001）

**解决方案：**
- 使用 `1000` 表示正常关闭
- 使用 `3000-4999` 范围内的代码表示自定义关闭原因
- 本实现使用 `3001` 表示心跳超时

### 问题：心跳超时频繁触发

**可能原因：**
1. 服务器未正确处理心跳消息
2. 网络延迟过高
3. `HEARTBEAT_TIMEOUT` 设置过小

**解决方案：**
1. 检查服务器是否正确返回 `pong` 响应
2. 增加 `HEARTBEAT_TIMEOUT` 的值
3. 检查网络连接质量

### 问题：连接仍然断开

**可能原因：**
1. 服务器主动断开连接
2. 防火墙或代理设置问题
3. 服务器未实现心跳处理

**解决方案：**
1. 检查服务器日志
2. 确认服务器支持心跳机制
3. 联系服务器管理员

## 最佳实践

1. **保持心跳间隔合理**
   - 不要过于频繁（避免服务器负担）
   - 不要过于稀疏（及时检测断开）

2. **超时时间设置**
   - 应该小于心跳间隔
   - 考虑网络延迟因素

3. **错误处理**
   - 心跳失败时记录日志
   - 连接断开时自动重连

4. **资源清理**
   - 连接关闭时清理所有定时器
   - 组件卸载时确保清理资源
